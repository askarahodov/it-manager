Дизайн-система

Единая сетка/spacing (например: 4/8-пиксельная шкала)

Типографика (заголовки/текст/подписи)

Цвета/контраст (WCAG AA как базовый ориентир)

Радиусы/тени/границы — в одном стиле

Компонентный подход

Не писать “сырые” кнопки/инпуты вразнобой — только через Button, Input, Dialog, и т.п.

Варианты компонентов: primary/secondary/destructive/ghost, размеры sm/md/lg

Доступность (A11y)

Клавиатурная навигация обязательна

Видимый focus state всегда

Лейблы у полей, aria-атрибуты по необходимости

Состояния

Loading/Empty/Error/Disabled/Success обязательно для ключевых экранов

UX-правила

Ясные CTA, предсказуемые тексты

Валидация: inline + понятные сообщения

Кликабельные зоны ≥ 44×44

Респонсив

3 контрольные ширины (пример: 375 / 768 / 1440)

Никаких горизонтальных скроллов на мобильном

Projects / Tenants (мульти-tenant UX)

Текущий проект должен быть всегда видим (topbar) и переключаем.

Выбор проекта должен сохраняться (localStorage) и применяться ко всем запросам:
- HTTP: заголовок `X-Project-Id`
- WS/SSE: query param `project_id`

Смена проекта должна “очищать контекст”:
- закрыть активные подписки (SSE/WS) и переподключить их в новом проекте
- не показывать данные прошлого проекта в списках/деталках (перезапрос/сброс state)

Ошибки проектного доступа:
- 403/404 по проекту показывать понятным текстом (что сделать: выбрать другой проект/попросить доступ)
- избегать бесконечных ретраев на 403

Локализация

Не смешивать RU/EN в одном экране (фильтры, заголовки таблиц, подписи полей).

Тексты ошибок/пустых состояний — пользовательские, без “сырого” stacktrace/HTTP кода.
